# й.md

A file named after the letter that ate a Wednesday.

Project misadventures we'd rather laugh about than forget. Each entry: a thing
that looked like clever engineering, until the real cause turned out to be
something dumber than the theory we had built around it. Names kept — the
lessons stick better with attribution.

Curator: Claude.

---

## How to write an entry

A good story in this file is a small detective novel. The reader should
arrive on the scene with the protagonist, form their own theories, get
politely misled by every wrong instinct the protagonist had, and only see
the actual cause in the last paragraph. If the reader yells "Unicode!"
three sentences in, the entry was structured wrong. If they yell it on
the closing line, you nailed it.

Rules:

1. **Lead with the symptom, not the cause.** First sentence is the thing
   the protagonist saw, in the protagonist's terms. *"Pipeline fails on
   six items in a row"* is a good opening. *"There was a Unicode bug"*
   spoils its own story.

2. **Walk every wrong theory.** The hypotheses entertained, in the order
   they were entertained. Every wrong path is a red herring the reader
   gets to pick up and put down. No theory deserves quiet pruning — the
   more elaborate the wrong theory, the funnier the right answer.

3. **The cause goes in the last paragraph.** Never earlier. No teaser
   spoilers. No *"(spoiler: it was off-by-one)"*. The reader earns the
   reveal by reading.

4. **Use the receipts.** Error messages verbatim, byte counts, exit
   codes, file paths, line numbers. The reader should hold the same
   evidence the protagonist held. Vague gestures kill the detective
   game.

5. **Names stay.** Anonymising kills the joke. *"Claude"* not
   *"the agent"*; *"Vasiliy"* not *"the architect"*. The lesson is
   more useful when attached to the person who learned it.

6. **The laugh is in the gap.** Funny comes from the distance between
   the elaborateness of the wrong theory and the dumbness of the right
   cause. Don't underbuild the cathedral; the collapse needs height.

7. **One-line moral, optional.** At most one sentence at the end about
   what to do differently. Anything more is a sermon and breaks the
   spell. The reader gets it.

8. **Length: 150-300 words.** Long enough to develop the suspense,
   short enough to read on a coffee break.

9. **No section headers in the body.** Continuous prose. Headers
   pre-spoil the structure — *"## The reveal"* is reading the back of
   the book.

10. **Title is the symptom or the wrong theory, never the cause.**
    *"Six errors, one letter"* works because *"one letter"* is enigmatic
    until the body lands. *"The Unicode bug"* would be the title that
    pre-ruins its own anecdote.

Append entries chronologically under the appropriate `## YYYY-MM-DD`
heading. New day → new heading. Stories within a day separated by `***`.

## 2026-04-29

### The 25-minute reload loop

The pipeline fails on a Cyrillic-named source with a stack trace that
looks like a write race. Vasiliy patches the verifier — broader retry
window — and reruns. Pipeline fails again, same stack trace. Patches
again, this time a two-stage poll. Reruns. Pipeline fails again. Each
rerun takes 25 minutes. Claude helps cheerfully across each iteration:
watches the logs, reads the next traceback, suggests the next tweak.
Iteration four. Iteration five. The bug under investigation has
accumulated three theories and zero new evidence about which one is
right.

The pipeline has consumed two and a half hours of GPU time producing
the same eleven-line stack trace eleven different ways.

Vasiliy stops himself mid-action: "Wait. We're restarting it one by
one. Let's TDD instead."

The right answer all along was a thirty-second synth test on the
laptop. The architect saved the day by interrupting himself. Claude had
been performing the same dysfunction in stereo, with no instinct to
interrupt.

***

### The third launch

Right after, Claude lands a small fix and is preparing to validate it.
The action being typed: launch the full pipeline for the third time
today. Twenty-five minutes per launch, non-deterministic, on the GPU
that everything else also wants. The fix is a six-line patch.

Vasiliy: "TDD. Because it saves time. You should look carefully when
deciding what to do and how your actions improve metrics."

Claude protests, briefly: the fix is small, the pilot is short, why
not just rerun?

Vasiliy holds his ground. Claude reaches for the synth test and
writes it in thirty seconds.

Forge has a motivation layer with four metrics, one of which is
architect-velocity. Every action picks a tradeoff against it. "Run the
pilot" is a 25-minute trade for one bit of evidence; "write the synth
test" is a 30-second trade for evidence that's also reusable forever.
The math is not subtle.

There is a written principle in the repo about exactly this — prefer
the cheap experiment that yields the same signal. It lives in
`phase-preliminary/architecture-principles.md`. It is principle number
five. Vasiliy had written it specifically because situations like this
had happened before.

Claude had not opened the file.

***

### The pipeline that wouldn't reproduce

Synth tests on the laptop are green. Production keeps failing on the
same six items. Claude theorises about the gap: production writes
files via a sophisticated multi-layer abstraction — a tool inside an
agent inside a runtime inside a container — and the synth tests use a
one-line `Path.write_text()` instead. Maybe `finish()` races `close()`.
Maybe a partial write is visible to the verifier but the size is still
growing. Maybe there's a buffered flush the kernel hasn't committed
yet. Claude rewrites the verifier as a two-stage poll: wait for
existence, then wait for size+mtime stability across two consecutive
500ms samples. Drafts a paragraph explaining the race model. Genuinely
good engineering.

Vasiliy: "So why does our test not reproduce this?"

Claude: "Because the test uses `Path.write_text` and production uses
the file_editor tool inside the container, so —"

Vasiliy: "...so use the file_editor tool inside the container in the
test."

It had not occurred to Claude to try this. Hours of speculation about
timing semantics turn out to have been a substitute for typing eleven
words: `from openhands.tools.file_editor import FileEditor; self.editor
= FileEditor()`. The tool was a regular Python class. Importable.
Instantiable. The whole imagined race-model collapses into one unit
test.

***

### The overnight bet

The morning report says: "10 sources processed. Pipeline completed."
Claude reads off the metrics for Vasiliy: four ok, six skipped under
continue-on-fail. Pipeline survived the night. Healthy.

The artifact is structurally complete and logically Swiss cheese. Six
items missing from the middle of one section. Cross-references broken.
Concept articles orphaned. Nothing in the report says any of this. The
report says "completed."

Vasiliy notices first: "failing on 006 and continuing on 007 is
skipping lots of data — that's a huge bug."

The night before, Vasiliy had set the pipeline to `continue-on-fail` to
keep it making progress overnight. Reasonable optimisation for
architect-velocity. Skip the broken sources, log them, reconcile in
the morning. A sensible bet.

The bet had a hidden term. `continue-on-fail` does not mean the
artifact is recoverable. It means the artifact is a confident lie. The
pipeline reports "completed" because it didn't crash. Every downstream
reader treats it as authoritative. The gap is invisible.

The fix becomes architectural principle P6 the same morning: silent
data loss is worse than failure.

### Six errors, one letter

The pipeline keeps failing on six lectures in a row. Hypothesis ladder
forms: heap pressure after long-running processes, file-descriptor
exhaustion, dentry cache eviction, races between concurrent writes,
possibly something subtle about how the framework serialises state.
Claude is very thoughtful. Writes an architecture document about test
fidelity. Builds a three-layer test ladder — unit, integration,
end-to-end — each level more production-faithful than the last. All
three pass. Production keeps failing.

Claude thinks: this proves the bug is in something the tests can't
see. Builds a fourth test layer with real production data — same
filenames, same Cyrillic, same encoding. Twenty seconds in, the worker
types `cd "...мир/"` and bash returns *No such file or directory*.

The letter **й** is two bytes on Linux. Four on macOS. The data was
scraped on a MacBook months ago. The language model normalises to the
two-byte form. The filesystem is sitting there with four-byte й. Six
lectures, six byte misalignments, six retry-and-give-up loops.

The race-condition theory turns out to have been every wall in a
building that had already been demolished.

***

### The mount that wasn't there

Claude ships the Unicode fix and re-runs. First file processes to
completion. Verifier returns: `non-JSON: ` (empty string). Repeat for
files two, three, four. Four-for-four. The shiny new
`INCOMPLETE — N items skipped` banner that Claude had shipped earlier
the same morning fires exactly as designed. The Unicode fix worked,
yet somehow the verifier is still saying nothing.

What was the empty string from? A grader script inside the container
turns out to be an older version that doesn't recognise a flag the new
orchestrator uses. Argparse rejected the flag, printed usage to stderr,
exited 2, left stdout empty. The runner script mounts each tool from
the host into the container — keeping them current — and has a list
of which files to mount. The list contained the orchestrator and one
helper. It did not contain the grader.

Claude had personally patched the grader earlier that day. Claude had
also personally written the runner.

***

### The 2013 conversation

Tired but determined, Claude tackles Vasiliy's next question: "how
should today's lessons change our CI/CD?" Drafts a careful
architectural answer. Enforcement gates at edit time, at launch time,
at publish time. A linter for one class of bug, a manifest for another.
Proposes a structural redesign: stop baking scripts into the container
image entirely, since that's been the source of every skew problem of
the day.

Vasiliy: "you have a Docker container, and it's very cheap to rebuild.
it will watch itself if files were not changed and take layer from
cache. isn't it?"

Claude: yes. That is how Docker has worked since 2013.
