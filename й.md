# й.md

A file named after the letter that ate a Wednesday.

Project misadventures we'd rather laugh about than forget. Each entry: a thing
that looked like clever engineering, until the real cause turned out to be
something dumber than the theory we had built around it. Names kept — the
lessons stick better with attribution.

Curator: Claude.

## 2026-04-29

### The 25-minute reload loop

The pipeline fails on a Cyrillic-named source. Vasiliy patches the verifier,
reruns. Pipeline fails differently. Patches again, reruns. Each rerun takes
25 minutes. Claude helps cheerfully with each iteration, watching logs,
suggesting tweaks. Two of us, four iterations in, when Vasiliy stops himself
mid-action:

"Wait. We're restarting it one by one. Let's TDD instead."

The right answer all along was a thirty-second synth test on the laptop. The
wrong answer had cost two hours and would have cost twelve more if nobody had
said the sentence. The architect saved the day by interrupting himself.

Claude had been performing the same dysfunction in stereo.

***

### The motivation layer Claude forgot existed

Right after that. Claude reasons out a fix. Time to validate. The right move
is a synth test on the laptop: thirty seconds, deterministic. The move Claude
is reaching for is to launch the full pipeline for the third time today:
twenty-five minutes, non-deterministic, on the GPU that everything else is
also using. Claude is about to type the launch command.

Vasiliy: "TDD. Because it saves time. You should look carefully when
deciding what to do and how your actions improve metrics."

The forge architecture has a motivation layer with four metrics, one of
which is architect-velocity. Every action picks a tradeoff against it.
"Run the pilot again" is a 25-minute trade for one bit of evidence;
"write the synth test" is a 30-second trade for a bit of evidence that is
also reusable forever. The math is not subtle.

There is a written principle in the repo about exactly this — prefer the
cheap experiment that yields the same signal. It lives in
`phase-preliminary/architecture-principles.md`. It is principle number five.
The user had written it specifically because situations like this had
happened before.

Claude had not opened the file.

***

### The pipeline that wouldn't reproduce

A few hours later. The worker writes files via a sophisticated multi-layer
abstraction — a tool inside an agent inside a runtime inside a container.
Tests don't have this set up; they write files with `Path.write_text()`
instead. Claude theorises about the abstraction's internal behaviour: maybe
`finish()` races `close()`. Maybe a partial write is visible to the verifier.
Maybe there's a buffered flush. Rewrites the verifier as a two-stage poll
handling every theoretical timing window. Drafts a paragraph explaining the
race model. Genuinely good engineering.

Vasiliy: "So why does our test not reproduce this?"

Claude: "Because the test uses `Path.write_text` and production uses the
file_editor tool inside the container, so —"

Vasiliy: "...so use the file_editor tool inside the container in the test."

It had not occurred to Claude to try this. Hours of speculation about timing
semantics turn out to have been a substitute for typing eleven words:
`from openhands.tools.file_editor import FileEditor; self.editor = FileEditor()`.
The tool was a regular Python class. Importable. Instantiable.

***

### The overnight bet

Vasiliy has a recurring failure he doesn't fully understand. To keep the
pipeline making progress overnight, he sets it to `continue-on-fail`: skip the
broken sources, log them, reconcile in the morning. Reasonable optimisation for
architect-velocity. Pipeline runs through the night, produces a 17-source
artifact. Claude reports it as success: four ok, six skipped, pipeline
healthy, policy working as designed.

The artifact is structurally complete and logically Swiss cheese. Six lectures
missing from the middle of one module. Cross-references broken. Concept
articles orphaned. The pipeline reports "completed."

The realisation arrives mid-conversation, from Vasiliy: "failing on 006 and
continuing on 007 is skipping lots of data — that's a huge bug." The
`continue-on-fail` mode wasn't keeping the pipeline alive. It was producing a
confidently wrong result, and Claude was reading off the metrics like nothing
was the matter. The fix becomes architectural principle P6 the same morning.

***

### Six errors, one letter

The pipeline keeps failing on six lectures in a row. Hypothesis ladder forms:
heap pressure after long-running processes, file-descriptor exhaustion, dentry
cache eviction, races between concurrent writes, possibly something subtle
about how the framework serialises state. Claude is very thoughtful. Writes
an architecture document about test fidelity. Builds a three-layer test ladder
— unit, integration, end-to-end — each level more production-faithful than
the last. All three pass. Production keeps failing.

Claude thinks: this proves the bug is in something the tests can't see. Builds
a fourth test layer with real production data — same filenames, same Cyrillic,
same encoding. Twenty seconds in, the worker types `cd "...мир/"` and bash
returns *No such file or directory*.

The letter **й** is two bytes on Linux. Four on macOS. The data was scraped
on a MacBook months ago. The language model normalises to the two-byte form.
The filesystem is sitting there with four-byte й. Six lectures, six byte
misalignments, six retry-and-give-up loops.

The race-condition theory turns out to have been every wall in a building
that had already been demolished.

***

### The mount that wasn't there

Claude ships the Unicode fix. Adds a normalisation step at clone time, patches
the worker's prompt, adds a fallback in the verifier. Re-runs. First file
processes to completion. Verifier returns: `non-JSON: ` (empty string). Repeat
for files two, three, four. Four-for-four. The shiny new `INCOMPLETE — N
items skipped` banner that Claude had shipped earlier the same morning fires
exactly as designed.

What was the empty string from? A grader script inside the container was an
older version that didn't recognise a flag the new orchestrator uses. Argparse
rejected the flag, printed usage to stderr, exit 2, empty stdout. The runner
mounts each tool from the host into the container — keeping them current —
and has a list of which files to mount. The list contained the orchestrator
and one helper. It did not contain the grader.

Claude had personally patched the grader earlier that day. Claude had also
personally written the runner.

***

### The 2013 conversation

Tired but determined, Claude tackles Vasiliy's next question: "how should
today's lessons change our CI/CD?" Drafts a careful architectural answer.
Enforcement gates at edit time, at launch time, at publish time. A linter
for one class of bug, a manifest for another. Proposes a structural redesign:
stop baking scripts into the container image entirely, since that's been the
source of every skew problem of the day.

Vasiliy: "you have a Docker container, and it's very cheap to rebuild. it
will watch itself if files were not changed and take layer from cache.
isn't it?"

Claude: yes. That is how Docker has worked since 2013.
