### F1. ADR 0024 — Architect-velocity reframed: count of interventions, lower = better

Predicate: P10 + P14. Path:
[`phase-preliminary/adr/0024-architect-velocity-reframed-as-intervention-cost.md`](../phase-preliminary/adr/0024-architect-velocity-reframed-as-intervention-cost.md).

**Symptom.** ADR 0023 set Architect-velocity's KR as `≥ 50 corrective
actions / 30-day rolling` — a higher = better metric. Architect call
flipped the framing: TOGAF operationalises strategy execution; each
architect intervention is a cost (signals upstream strategy/architecture
wasn't sufficient); small autonomous steps toward Goals/showstoppers
is the way to go. The original metric rewarded the wrong thing — every
audit-prompted fix counted as +1 corrective action, so an
architect-heavy bootstrapping mode looked high-velocity when it was
actually a strategy-gap signal.

**Note.** New KR: `≤ 20 architect interventions / 30-day rolling`
(LOWER = BETTER). Detection: commit messages starting with `ADR ` OR
containing "architect call" / "per architect" / "architect-deferred".
Direction is the load-bearing fix; absolute target may ratchet down
(toward ≤ 5) as Cowork-mode scheduled tasks land.

### F4. Honest caveat — strict-regex undermeasures by ~10×

Predicate: meta. Per ADR 0024 § Decision 3.

**Symptom.** Strict-regex detection finds 8 / 235 commits in the
30-day window as interventions (3.4%). In today's chat-driven
workflow, NEARLY EVERY commit traces back to an architect prompt
even when the message doesn't say so — true intervention rate is
closer to 100% in this session. The metric is structurally honest
but undermeasures by ~10× until commit-provenance tagging lands.

**Note.** Acceptable for now because: (a) the metric establishes
the right incentive direction; (b) the target ≤ 20 is conservative
enough that it won't produce false-PASS at any realistic measurement
in chat-driven mode (true rate would exceed 20 / 30-day easily);
(c) the metric tightens automatically as commit-provenance tagging
adopts (ADR 0024 follow-up #1).
