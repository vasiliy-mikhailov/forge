# D8 — Retrieval-augmented dedup: implementation spec

**Status:** Draft
**Date:** 2026-04-26
**Driven by:** ADR 0010
**Predecessor:** D7-rev4-v2 (linear `prior_claims_json`, last viable rev)

---

## Goal

Replace the `prior_claims_json` linear scan in skill v2 ritual with a
retrieval-augmented mechanism that scales to ≥200 sources without
context-window exhaustion or recall collapse.

Two parallel deduplications must work end-to-end:
1. **Claims dedup** — `idea-classifier` decides NEW vs REPEATED via top-K
   retrieval over a vector index of prior claims, then LLM-judges the
   shortlist.
2. **Concepts dedup** — `concept-curator` checks whether a kebab-slug or
   its definition is semantically equivalent to an existing concept
   before creating a new file.

---

## Hypothesis

H1. Retrieval-then-judge gives equal-or-better REPEATED recall vs current
linear LLM scan, because (a) the LLM no longer fights context dilution,
(b) embedding-based candidate selection catches synonym paraphrases that
keyword search misses.

H2. Wall-time per classifier task drops by ~50 % because input context
shrinks from ~35 K (transcript + 30 K prior_claims_json) to ~5 K
(transcript-trimmed + ~3 K candidates).

H3. Concept count after dedup at the 14-source scale is 25-35 % lower
than no-dedup baseline, with no information loss (validated by manual
review of merged concepts).

H4. Embedding index size grows linearly (~250 KB per source). At 200
sources ≈ 50 MB, well within git-LFS comfort zone.

## Architecture

### Components

```
wiki/
  data/
    embeddings/
      manifest.json            # {model_name, model_version, indexed_at, source_count}
      claims.sqlite            # virtual table claims_vss (cosine)
      concepts.sqlite          # virtual table concepts_vss (cosine)
  skills/
    benchmark/
      requirements.txt         # NEW deps: sentence-transformers, sqlite-vss
      scripts/
        embed.py               # NEW: encode text → vector (CLI + library)
        rebuild_index.py       # NEW: full rebuild from .md files
        update_index.py        # NEW: incremental, post-source-commit
        find_similar_claims.py # NEW: replaces get_known_claims.py
        find_similar_concepts.py # NEW: concept dedup helper
        get_known_claims.py    # KEPT for backwards-compat, deprecation warning
```

### Data model

**`claims.sqlite` schema:**
```sql
CREATE TABLE claims (
  id          INTEGER PRIMARY KEY,
  source_slug TEXT NOT NULL,            -- 'Психолог-консультант/005…/000 Вводная'
  claim_idx   INTEGER NOT NULL,         -- position in source's ## Claims list
  claim_text  TEXT NOT NULL,
  marker      TEXT NOT NULL,            -- 'NEW' | 'REPEATED' | 'CONTRADICTS_FACTS'
  url         TEXT,
  thematic_category TEXT,               -- from frontmatter or inferred
  notes       TEXT,                     -- ## Claims notes paragraph if present
  indexed_at  TEXT NOT NULL,            -- ISO 8601
  UNIQUE (source_slug, claim_idx)
);
CREATE VIRTUAL TABLE claims_vss USING vss0(
  embedding(768)                        -- e5-base output dim
);
-- claims_vss.rowid == claims.id (1:1 mapping)
```

**`concepts.sqlite` schema:**
```sql
CREATE TABLE concepts (
  slug          TEXT PRIMARY KEY,
  title         TEXT,                   -- from H1 in concept .md
  definition    TEXT,                   -- ## Definition paragraph(s) joined
  introduced_in TEXT,                   -- source_slug
  touched_by    TEXT,                   -- JSON array of source_slugs
  indexed_at    TEXT NOT NULL
);
CREATE VIRTUAL TABLE concepts_vss USING vss0(
  embedding(768)
);
```

### Embedding pipeline

`embed.py` exposes:
```python
from sentence_transformers import SentenceTransformer
_model = SentenceTransformer('intfloat/multilingual-e5-base')

def encode_claim(text: str) -> list[float]:
    # e5 expects "query: ..." prefix for retrieval queries
    # and "passage: ..." for indexed documents
    return _model.encode(f"passage: {text}", normalize_embeddings=True).tolist()

def encode_query(text: str) -> list[float]:
    return _model.encode(f"query: {text}", normalize_embeddings=True).tolist()
```

For concepts, both `slug` and `definition` are encoded; we store
`encode_passage(f"{slug}\n\n{definition}")`.

### Retrieval API

`find_similar_claims.py`:
```bash
python3 find_similar_claims.py \
  --claim "<text>" \
  [--module <path>] \
  [--course <name>] \
  [--k 5] \
  [--threshold 0.65]
```

Output (stdout):
```json
{
  "candidates": [
    {
      "source_slug": "Психолог-консультант/005…/000 Вводная",
      "claim_idx": 12,
      "claim_text": "В лимбической системе десятки ядер выживания",
      "marker": "NEW",
      "thematic_category": "neuroanatomy",
      "similarity": 0.91
    }
  ],
  "k_requested": 5,
  "k_returned": 1,
  "below_threshold": 0,
  "model": "intfloat/multilingual-e5-base"
}
```

Exit codes:
- 0 = success (candidates may be empty)
- 1 = index missing or corrupt
- 2 = bad arguments

`find_similar_concepts.py`:
```bash
python3 find_similar_concepts.py \
  --slug-or-text "social-instinct" \
  [--k 3] \
  [--threshold 0.85]
```
Output:
```json
{
  "candidates": [
    {"slug": "social-need", "title": "Социальная потребность",
     "similarity": 0.89}
  ]
}
```

### Index updating

`update_index.py --source <slug>`:
1. Read `wiki/data/sources/<slug>.md`.
2. Parse `## Claims — provenance and fact-check` section.
   - Numbered list: `<n>. <text> [<marker>]\n<notes>?\n— <url>?`
   - Use existing parser from `bench_grade.py`.
3. For each claim, encode + insert into both `claims` and `claims_vss`.
4. Read frontmatter `concepts_introduced`, for each:
   - Read `wiki/data/concepts/<slug>.md`.
   - Encode title+definition, upsert into `concepts` and `concepts_vss`.
5. Update `manifest.json` with new source count and timestamp.

Idempotent: if rows already exist with same `(source_slug, claim_idx)`,
delete and re-insert.

`rebuild_index.py [--force]`:
1. Drop and recreate both .sqlite files.
2. `find wiki/data/sources -name '*.md'` → for each, call `update_index.py`.
3. Same for concepts.
4. Print summary: total claims indexed, total concepts indexed, wall time.

`embed_index.py --query <text> --collection claims|concepts`:
Debug helper. Just dumps top-K candidates with similarity scores.

### Sub-agent prompt changes

**`idea-classifier` system prompt diff:**

Before (D7-rev4-v2):
```
Each task message contains:
  - claim
  - lecture_transcript
  - prior_claims_json: output of get_known_claims.py
```

After (D8):
```
Each task message contains:
  - claim
  - lecture_transcript

Step 1: Run via terminal:
  python3 wiki/skills/benchmark/scripts/find_similar_claims.py \
    --claim "<claim>" --k 5 --threshold 0.65

This returns up to 5 semantically-similar prior claims. Treat the
output as the *only* set of REPEATED candidates — do NOT fabricate
matches outside this list.

Step 2: For each returned candidate, read its `claim_text`.
  - If any candidate is a true paraphrase of the new claim:
      output `REPEATED from <source_slug of best match>`.
  - Otherwise: `NEW`.

Step 3: Pick thematic_category as before.

Output ONE LINE: `<verdict> | category=<theme>`.
```

The classifier now has terminal access (was `tools=[]`); minor prompt
expansion to handle JSON parsing (~50 added lines).

**`concept-curator` system prompt diff:**

Add new step before file creation:
```
Step 0 (NEW): Before creating <concept_slug>.md, check for an existing
near-duplicate:
  python3 wiki/skills/benchmark/scripts/find_similar_concepts.py \
    --slug-or-text "<concept_slug>" --k 3 --threshold 0.85

If a candidate has similarity ≥ 0.85, DO NOT create a new file. Instead:
  1. Read the existing <existing_slug>.md
  2. Append <source_slug> to its `touched_by:` list (if not present)
  3. Output `concept <existing_slug> reused (similarity=<X>)` and finish.

Otherwise proceed to create as before.
```

**`source-author` orchestrator prompt diff:**

Remove Step B ("Get known claims via get_known_claims.py").
Replace with:
```
### B. (gone — classifier now retrieves on-demand per claim)
```

Move retrieval responsibility into idea-classifier (where it belongs;
it's the agent that does the comparing).

## Test ladder

Each step has a hard pass criterion. Runs incrementally on `synth/`
fixtures first, then on a 7-source pilot, then 14-source scale test.

### Step 0 — Python-loop top-orchestrator (must precede retrieval work)

D7-rev4-v2 production showed top-orchestrator collapses at ~5 sources due
to accumulated task() result history (8.93 M cumulative input tokens).
Fix: replace LLM-conversation top-orch with Python `for` loop that
spawns a fresh `Conversation(...)` per source.

**Refactor `run-d8-pilot.py`:**

```python
def main():
    branch, served = setup_workspace()
    sources = list_sources()
    setup_agents()

    state = []
    for (n, raw_path, target_path) in build_inputs_text(sources):
        # Fresh conversation per source — no carry-over context
        llm = LLM(...)
        main_agent = Agent(llm=llm, tools=[Tool(name="task_tool_set")])
        conv = Conversation(agent=main_agent, workspace=str(WORKDIR), ...)

        msg = (f"Process source N={n}. raw_path={raw_path}. "
               f"target_path={target_path}. "
               "Use the `task` tool with subagent_type='source-author'. "
               "Wait for `done` ack. Then finish.")
        conv.send_message(msg)
        conv.run()

        v = verify_source(n)
        if v.get("verified") != "ok":
            state.append({"n": n, "verify": v, "stopped": True})
            break

        commit = commit_and_push_per_source(n, slug, branch, first=(n == 0))
        update_index(n, slug)  # NEW — D8 retrieval index update
        state.append({"n": n, "verify": v, "commit": commit})

    write_bench_report(state, branch, partial=any(s.get("stopped") for s in state))
```

**TDD validation (in `step6_orchestrator.py`):**

1. **Synth GREEN** with same skill v2 ritual on 4 synth sources:
   - 4/4 verified=ok
   - REPEATED ≥ 2, CF=2, claims ≥ 20
   - canonical concept template OK

2. **Per-source top-orch context bound (HARD ASSERT)**:
   ```python
   assert n_events_per_source <= 30
   assert input_tokens_per_source <= 100_000
   ```
   These bounds encode the architectural invariant. Any future change
   that re-introduces a single-Conversation top-orch will fail this
   smoke and block merge.

3. **Linearity check**: cumulative top-orch input tokens scale O(N) of
   per-source, not O(N²). Concretely: each source's input ≤ 2× the
   running average. If any iteration violates, we have a Conversation
   re-use bug.

4. **Production validation**: run on D7-rev4-v2 same 7-source module 005,
   expect **7/7 verified=ok** (vs. current 5/7) and per-source bounds hold.

**This is now Step 0** — gated before any retrieval work, because we
can't bench D8 retrieval if the orchestrator can't even reach sources 5-6.

## Risks & mitigations

| Risk | Likelihood | Mitigation |
|---|---|---|
| Embedding model under-recalls Russian paraphrases | M | Calibrate threshold on synth, add BM25 hybrid as fallback |
| `sqlite-vss` package install issues on prod server | L | Pre-build wheel in Docker image, document fallback to ChromaDB |
| 50 MB sqlite blob in git annoys collaborators | M | Use git-LFS; alternatively, .gitignore embeddings/ and rebuild on first checkout (~5 min added to onboarding) |
| Concept dedup creates false-merges (different concepts → same slug) | M | Threshold 0.85 (tight); manual audit Step 7; Easy revert: delete from `touched_by:` and recreate file |
| Classifier with terminal access misuses it | L | Tight system prompt; tool whitelist enforced by AgentDefinition |
| Cross-module REPEATED creates noise (claim in module 004 about Maslow incorrectly tagged as REPEATED in module 005's intro about Maslow) | M | Explicit thematic_category match required for cross-module REPEATED; same-module REPEATED has no extra check |
| LLM ignores retrieval results, hallucinates REPEATED | L | TDD validates exit-code/format strictness; classifier fails-soft (returns NEW if format breaks) |

## Out of scope (explicit)

- **Full-text search UI** — same index will support it later, but UI is D9+.
- **Multi-language support** — wiki is Russian-first; e5-base handles it
  natively. English claims (if any) work fine but not optimized.
- **Citation graph** — "which concepts depend on which other concepts"
  via cross-references; separate spec.
- **Auto-merge of duplicate sources** — out of scope; manual decision.
- **Re-grading of historical sources** when new claims arrive that
  retroactively make older [NEW] claims actually [REPEATED] from the
  newer one — this is "Stratified [REPEATED] semantics" and is D9+.
  For D8, retrieval is forward-only: source N can mark REPEATED only
  from sources < N.
